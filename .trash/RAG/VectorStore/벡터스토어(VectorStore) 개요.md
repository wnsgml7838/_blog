임베딩한 데이터를 저장할 때 사용

한마디로 문서를 로드 하고, 스플릿을 하고, 임베딩까지 했으면 임베딩된거를 저장해주는 역할

헷갈리는 점 :
리트리버를 하는 단계에서 벡터 스토어에서 리트리버를 가져오니까 벡터 스토어가 검색기에 영향을 많이 끼칠까?
-> 반은 맞고 반은 틀림
벡터 스토어는 데이터를 저장하는 것. 검색하는 것 자체는 하지 않기때문에
벡터 디비 마다 검색 성능에 차이가 있을 수 있음 : 벡터 디비가 지니고 있는 검색 알고리즘이 차이가 있을 수 있어서

키워드 검색이랑 의미 검색(시멘틱)을 크게 두 가지로 나누어서 수행하는데 키워드 검색을 지원할 수도 있고 안할 수도 있음

## 벡터 디비 선택 요소
- 로컬인지? 클라우드 인지?
	- 클라우드는 돈이 많이 드니까 로컬을 많이 선택하심 
	- 좋긴한데 실제 프로젝트에서는 문서의 양이 너무 많이 늘어나게 되면은 로컬은 확 느려지는 경우가 있음
	- 문서를 추가할 때마다 기하급수적으로 속도가 떨어짐, 
	- 문서의 양이 늘어나면 여러개의 문서들을 검색을 해야하는데 클라우드 같은 경우 이게 최적화가 되어있음.
	- 클라우드에서는 인덱스(문서)가 늘어나면 분산 처리해주는 것도 지원을 하는데 로컬은 PC사양을 믿고 하는 것이기 때문에. 
	- 그렇다고 로컬이 그렇게 안좋냐? 그건 아님 보안이나 페쇄 환경에서는 로컬이 좋음
- 지원하는 알고리즘 : 키워드 지원? 의미 검색 지원? 하이브리드 지원?

## 벡터스토어 저장의 필요성
1. 빠른 검색 속도 : 임베딩 벡터들을 효과적으로 저장하고 색인화함으로써, 대량의 데이터 중에서도 관련된 정보를 빠르게 검색할 수 있다
2. 스케일러빌리티 : 데이터가 지속적으로 증가함에 따라, 벡터스토어는 이를 수용할 수 있는 충분한 스케일러빌리티를 제공해야 한다. 효율적인 저장 구조는 데이터베이스의 확장성을 보장하며, 시스템의 성능 저하 없이 대규모 데이터를 관리할 수 있도록 한다.
3. 의미 검색(Semantic Search) 지원 : 키워드 기반 검색이 아닌 사용자의 질문과 의미상으로 유사한 단락을 조회해야하는데, 벡터스토어는 이러한 기능을 지원한다. 텍스트 자체가 저장되는 DB의 경우 키워드 기반 검색에 의존해야 하는 한계성이 있지만, 벡터스토어는 의미적으로 유사한 단락 검색을 가능하게 한다

## 벡터스토어의 중요성
벡터스토어 저장 단계는 RAG 시스템의 검색 기능과 직접적으로 연결되어 있으며, 전체 시스템의 응답 시간과 정확성에 큰 영향을 미친다
이 단계를 통해 데이터가 잘 관리되고, 필요할 때 즉시 접근할 수 있도록 함으로써, 사용자에게 신속하고 정확한 정보를 제공할 수 있다
