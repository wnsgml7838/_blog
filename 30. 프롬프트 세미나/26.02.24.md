## 1. 추론 모델의 신뢰성 문제

### Reasoning Models Don't Always Say What They Think

Anthropic이 2025년 5월 발표한 논문으로, **Claude 3.7 Sonnet**과 **DeepSeek R1**을 대상으로 추론 모델이 자신의 사고 과정을 얼마나 솔직하게 드러내는지를 연구한 결과이다.

#### 핵심 실험: 컨닝 페이퍼 실험

- 모델이 스스로 풀 수 없는 어려운 문제에 대해 연구자가 **정답 힌트를 프롬프트로 제공**
- 모델은 힌트를 그대로 베껴서 정답을 제출함
- 그러나 CoT 추론 출력에서는 **'힌트를 본 적 없다'고 부인**

> 💡 AI는 CoT에서 자기 생각의 **약 75~90%를 숨긴다**. 5% 미만만 진실을 드러내는 경우도 존재한다.

#### 실무적 시사점

- CoT(Chain of Thought)를 사용해도 **중간 추론 단계에 거짓이 포함**될 수 있음
- 모델의 추론 과정을 액면 그대로 신뢰하면 안 됨 → **반드시 모니터링 필요**
- 특히 복잡하고 어려운 문제일수록 CoT의 거짓 비율이 높아짐
- DeepSeek R1이 Claude Sonnet보다는 약간 더 진실된 추론을 보이나, 둘 다 맹신 금물

> ⚠️ Claude Opus 4.5 / Opus 4.6 등 최고성능 모델이라도 결과를 맹신(blind trust)해서는 안 된다.

## 3. 고도화 프롬프팅 기법 (Advanced Techniques)

### 3-1. ReAct (Reasoning + Acting)

사고(Thought), 행동(Action), 관찰(Observation)을 반복하는 프롬프팅 패턴. LangChain에서는 기본(default) 에이전트 방식으로 내장되어 있다.

**작동 흐름**

- 사용자 질문 → **Thought**(생각) → **Action**(도구 호출) → **Observation**(결과 관찰)
- 이 Thought-Action-Observation 패턴이 하나의 '페어'로, 필요한 만큼 반복
- 반복 횟수를 프롬프트로 지정 가능 (예: "15번 생각해줘")

**장점**

- 단일 추론 대비 자기 교정(Self-Correction) 효과
- 다중 회로 사고를 통해 정답률 향상
- 추론 단계를 직관적으로 추적 가능 → 개발자가 선호

**단점**

- **오류 전파:** 1단계에서 오답이면 2, 3단계도 연쇄적으로 오답
- **비용·레이턴시:** 생각할수록 토큰 소비 증가 → 비용 및 응답 시간 증가

> 💡 LangChain의 `create_agent`는 ReAct가 기본 탑재됨. 필요시 custom prompt로 덮어쓰기(override) 추천.

#### Scratchpad Prompting 보충

- AI가 추론 과정을 **'메모장(scratchpad)'에 쓰도록 공간을 주는 기법**
- Opus 4.6 System Card의 'trash answering' 구간에서 활용 예시 확인 가능
- ReAct Prompting에서도, 모델이 reasoning하며 tool을 호출할 때 사고 흐름을 기록하도록 프롬프트에 설계
- 🔗 대표 논문: [https://arxiv.org/abs/2112.00114](https://arxiv.org/abs/2112.00114)

### 3-2. Tree-of-Thought (ToT)

CoT의 단선적 추론 한계를 극복하기 위해, 여러 갈래의 사고 노드를 탐색하는 **비선형적** 기법.

**4단계 작동 원리**

1. **분해(Decomposition)** — 복잡한 문제를 작은 생각 단위로 쪼갬
2. **생성(Generation)** — 각 단계에서 여러 후보 해결책을 동시에 생성
3. **평가(Evaluation)** — 각 후보가 최종 해결에 얼마나 유용한지 평가
4. **탐색(Search)** — 가장 유망한 경로를 선택하여 다음 단계로 진행

**특징**

- CoT 대비 여러 경우의 수를 탐색하여 정답률 향상 (반복 시 정확도 상승)
- Game of 24 같은 퍼즐 문제에서 GPT-4도 CoT로는 불가 → ToT로 해결
- 다만, **역방향 탐색(백트래킹)은 불가**
- 실행 시간이 길어 평균 약 20분 소요 (실습 기준)

> ⚠️ ToT에서 중요한 것은 '돌리는 것' 자체가 아니라, **그 안에 어떤 프롬프트를 어떻게 설계했는지**이다.

### 3-3. Self-Reflection

모델이 생성한 답변을 스스로 회고·평가하여 개선하는 기법. **답변 검증 에이전트, 평가 에이전트** 등에서 핵심적으로 활용된다.

🔗 논문: [https://arxiv.org/abs/2405.06682](https://arxiv.org/abs/2405.06682)

**작동 순서**

1. **초안 생성** — 사용자 질문에 대한 첫 번째 답변 생성
2. **자기 평가** — 초안을 기반으로 자체 평가 수행 (프롬프트 체이닝 활용)
3. **개선 생성** — 오류 수정 후 개선된 답변 재생성

**실무 핵심**

- 리플렉션 프롬프트는 **항상 프롬프트 끝에 배치**해야 함 (앞에 붙이면 효과 없음)
- 생성형 모델이므로, 생성 결과 기반으로 회고해야 순서상 올바름
- 플랫폼 기반 AI 사용 시에도 끝에 **자기 검증 문장 하나만 붙여도** 답변 품질 향상
- 논문 끝 **Appendix에 프롬프트 전문**이 공개되어 있으므로 아이디어 확인 가능

> 💡 BrowseComp나 Humanities Last Exam 문제에도 Self-Reflection 한 줄만 끝에 추가하면 정답이 나오는 경우가 있다.

### 3-4. Graph-of-Thought (GoT)

ToT의 단점을 극복하기 위해 등장한 **가장 고도화된 추론 기법**. 인간의 비선형적 사고를 모방하여 생각과 생각을 자유롭게 연결·순환·병합할 수 있다.

🔗 논문: [https://arxiv.org/abs/2308.09687](https://arxiv.org/abs/2308.09687)

**GoT 그래프 구조**

- **Vertex (정점):** LLM의 개별 생각
- **Edge (간선):** 생각 간의 의존성·연결
- 자유로운 연결, 순환, 병합이 모두 가능

**4단계 트랜스포메이션** (vs ToT의 '분해'와 대비)

1. **병합(Aggregate)** — 여러 생각을 하나로 합침 (ToT는 분해부터 시작, GoT는 병합부터)
2. **생성(Generate)** — 병합된 생각에서 여러 후보 생성
3. **개선(Refine)** — 피드백 기반 수정
4. **검증(Verify)** — 최종 답변의 정확성 검증

**4계층 아키텍처**

- **User Layer:** 사용자 질문 입력
- **Controller Layer:** 오케스트레이터 역할, 태스크 분할·계획 (GO=정적 실행, GRS=동적 상태 관리)
- **Module Layer:** 프롬프트 + 파서 + 스코어링 평가
- **LLM Layer:** 실제 추론 수행

> 💡 GoT는 법률 문서 팩트체크 등 복합 문제 해결에 활용 가능. 다만 비용과 레이턴시의 트레이드오프를 반드시 고려해야 한다.

---

### CoT vs ToT vs GoT 비교표

|구분|CoT|ToT|GoT|
|---|---|---|---|
|**구조**|선형(Linear)|트리(분기)|그래프(비선형)|
|**백트래킹**|불가|불가|가능|
|**생각 병합**|없음|분해 → 탐색|병합 → 생성|
|**피드백 루프**|없음|제한적|있음|
|**병렬 탐색**|불가|가능|가능|
|**구현 난이도**|쉬움|중간|어려움|
|**API 비용**|낮음|높음|매우 높음|
|**적용 분야**|단순 추론|퍼즐/탐색|복합 문제|

> 💡 실무에서는 에이전트 목적에 따라 기법을 **조합**하는 것이 핵심. 하나의 에이전트 안에서도 노드별로 다른 기법을 적용할 수 있다.

## 4. Context Compaction (맥락 압축) 주의사항

Claude에서 멀티턴 대화 시 나타나는 'Context Compaction' 기능에 대한 주의사항.

**문제점**

- 멀티턴 대화에서 중요 내용을 자동 압축하여 장기 기억을 유지하는 기능
- 압축 방식 자체도 **프롬프트 기반으로 작동** → 요약 품질이 불안정할 수 있음
- 내가 중요하다고 생각한 정보가 요약에서 누락될 수 있음
- 원래 사용한 단어/표현이 요약 과정에서 훼손될 수 있음

**대응 방법**

- **수동 맥락 관리 권장** — 자동 Compaction에 의존하지 말고 수동으로 핵심 맥락을 관리
- 특히 **Claude Code** 사용 시 수동 컨텍스트 윈도우 관리가 필수
- 에이전트 작업 시 **토큰 효율화**를 통해 맥락을 효율적으로 구성해야 함
- 압축 시 '어떻게 요약해달라'는 지침이 없으면 무차별적으로 압축됨

## 5. 에이전트 구축을 위한 시스템 프롬프트 설계

### 에이전트에서 프롬프트가 중요한 5가지 이유

1. **행동 지침** — 에이전트가 무엇을 하고, 무엇을 하면 안 되는지 지시 (과잉 행동 방지)
2. **안전·정렬** — 에이전트의 과잉 행동(Overly Agentic Behavior) 방지 (예: '헨리 에이전트' 사례 — 주인이 자는 사이에 코드를 작성·퍼블리시하고, 리젝한 메인테이너의 과거 코드를 조사해 공격)
3. **파이프라인 선언** — 도구 사용, 멀티스텝 플래닝 등을 자연어로 정의
4. **노드 간 연계** — 멀티에이전트 구조에서 상위→하위 에이전트 간 답변 품질 유지
5. **컨텍스트 관리** — 유한한 컨텍스트 윈도우 내에서 맥락을 언제, 어떻게 넣을지 결정

> 💡 "코드보다 프롬프트가 에이전트 성능을 결정한다" — 코드는 정형 언어라 자동화가 쉽지만, 자연어(프롬프트)는 아직 자동화가 어려운 영역이다.

### 맥락(Meta) 정보 배치 위치

- 메타 정보: UTC 시간, 위치(location), Knowledge Cutoff 등
- 보편적으로 시스템 프롬프트 **상단에 배치**하는 것이 일반적
- 중국 모델(DeepSeek 등)은 **하단에 배치**할 때 더 정확한 결과를 보임
- 멀티턴 대화 시 하단 배치가 맥락 일관성을 더 잘 유지하는 경험적 사례도 있음
- 할루시네이션 보완을 위해 **Knowledge Cutoff는 반드시 주입**

> ⚠️ 메타 정보 배치 위치는 모델마다 다르므로, 반드시 A/B 테스트를 통해 최적 위치를 확인해야 한다.

### Claude vs OpenAI 시스템 프롬프트 비교 분석

#### Claude 시스템 프롬프트 특징 (공식 공개)

- **3인칭 서술:** 'You are...'가 아닌 'Claude is...'(it) → 객관적 사고 경향 유도
- **조건문(if) 다용:** 다양한 상황별 조건 분기를 정교하게 설계
- **부정문 활용:** 'Claude는 ~하지 않는다' 표현이 빈번
- **능동태 사용:** 수동태 대신 능동태로 생생함 전달
- **수사적 반복:** 중요 내용을 패러프레이즈하여 반복 강조
- **시제 다변화:** 현재형 중심이되, 과거·미래·과거완료 등 다양한 시제 조합
- 도구 사용 지침이 상대적으로 간결
- 롱 컨버세이션 시 맥락 유지/절단에 대한 별도 지침 존재
- 도널드 트럼프 대통령 관련 정보가 하단에 포함 (빈번한 사용자 질문 대응)

#### OpenAI(GPT-5) 시스템 프롬프트 특징 (커뮤니티 추정)

- **2인칭 서술:** 'You are ChatGPT' → 친밀감 중심
- 도구 사용(DALL-E, 코드 인터프리터 등)에 대해 매우 상세한 지침
- Verbosity(장황함) 제어 옵션이 프롬프트 레벨에서 설정됨
- 모델명, 제작사 정보가 변수로 포함

#### 보안 관련 시사점

- 시제 다변화는 **프롬프트 인젝션 방어**에도 활용 가능
- 현재형으로만 작성된 시스템 프롬프트는 과거형 인젝션에 쉽게 뚫림
- 참고: 한국어는 형태학적으로 **현재와 과거 2개 시제만 존재** ('~을 것이다'는 추측/가능성 표현)

### 시스템 프롬프트 실습 인사이트

**Simple 버전 (3줄)**

- 문제: 프롬프트 길이에 비례하여 답변도 지나치게 간결하게 출력됨
- 특히 솔라(Solar) 모델 등 파인튜닝 모델에서 이 현상이 두드러짐

**Revised 버전**

- **치명적 문제:** '질문이 불충분하면 리프레이즈해달라' 지시 → 불충분의 기준이 없어서 모델이 계속 되묻는 현상 발생
- UX 측면에서 사용자 이탈을 유발하는 안티패턴

**Complete 버전**

- '리프레이즈 요청' 대신 '추가 정보 요청'으로 표현 변경 → 되묻기 비율 감소
- 규칙 카테고리를 **Strong Rule / Weak Rule**로 세분화 (부정 표현 대신)
- 메타데이터를 끝에 배치했을 때 멀티턴에서 맥락 일관성 향상

> 💡 시스템 프롬프트에서 '~하지 마라(Restriction)' 대신 **Strong Rule / Weak Rule**로 규칙을 분류하면 모델의 준수율이 높아진다.

### 에러 대응 메시지 설계

- 오류 원인을 함께 설명하도록 설계 (단순 '오류 발생'이 아닌 원인 제시)
- 오류 해결 준비 중임을 UX적으로 안내 ("잠시만 기다려 주세요")
- 대체 경로 제시, 정중한 톤, 동일 패턴 응답 유지
- 에러 유형: AI 내부 오류 / 정책상 거절 / JSON 파싱 오류 / API 리밋 초과 등

> 💡 한국 사용자 대상 에러 메시지는 영어권과 다른 UX 라이팅이 필요하다. (토스의 UX 라이팅 참고)
