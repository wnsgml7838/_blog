## 리서치 1. AI의 생물학 : LLM 내부를 들여다보다

- **논문** : [On the Biology of a Large Language Model](https://transformer-circuits.pub/2025/attribution-graphs/biology.html) (2025.03)
- **연구팀** : Anthropic
- **대상 모델** : Claude 3.5 Haiku
- **핵심 방법** : Attribution Graphs 도구를 활용해 10가지 모델 행동을 심층 분석. 특정 입력이 출력으로 변환되는 과정에서 "features" 활성화를 시각화함.

모델이 어떤 단어에 어떻게 연관되어 있고, 토큰이 다음 토큰을 어떻게 생성하는지 — 지금까지 블랙박스였던 내부 작동 원리를 이제는 시각적으로 확인할 수 있게 되었다.

### 핵심 발견 요약

1. 모델의 **2단계 추론**을 목격하다
2. AI는 시를 쓸 때 **"미리 계획"**한다
3. **보편적 사고 언어** — 다국어 언어 처리의 비밀

### 이 연구가 왜 중요한가? — 해석 가능성(Interpretability)

LLM은 인간이 직접 프로그래밍한 것이 아니라, 대량의 데이터로 학습하며 스스로 문제 해결 전략을 형성한다. 그 전략은 수십억 개의 계산 속에 암호화되어 있어, 개발자조차 정확히 이해할 수 없었다.

Anthropic은 이를 **생물학**에 비유한다. 생물학자가 수십억 년 진화를 통해 만들어진 생물체를 현미경으로 관찰하듯, AI 연구자들이 LLM 내부를 "AI 현미경"으로 들여다보는 것이다.

- MIT Technology Review가 **기계적 해석 가능성(Mechanistic Interpretability)**을 2026년 10대 기술로 선정
- Opus 4.6 System Card에 해석 가능성 기법을 처음으로 안전성 평가에 적용

```
Dallas가 속한 주를 먼저 파악하고, 그 주의 수도를 답변해주세요.
각 단계를 명시적으로 보여주세요.
```

**복잡한 분석 작업에의 적용 (예: 법적 문서 검토) :**

워크플로우를 프롬프트에 명시한다.

```
1. 계약 유형 파악
2. 핵심 조항 추출
3. 위험 요소 평가
위 순서대로 단계별로 분석해주세요.
```

> 💡 특히 안전한 로컬 LLM(비추론 모델)을 선호하는 기업 환경에서, 고성능 추론 모델 대신 보안·온프레미스 모델을 사용할 때 이 전략이 더 효과적이다.

### 핵심 발견 2. AI는 시를 쓸 때 미리 계획한다

**가설** : 모델은 과제 수행 시 계획하지 않는다 — 다음에 올 토큰의 확률 값에 따라 순차적으로 생성할 뿐이다.

**실험 결과** : 줄바꿈 토큰이 일종의 **계획 지점(planning point)**으로 기능한다.

"He saw a carrot ..."로 시작하면, 모델은 줄바꿈 토큰에서 `rabbit`이나 `habbit` 같은 후보 단어를 **먼저 활성화**한 후, 그 단어를 향해 문장을 생성한다.

랩에서 끝에 라임을 맞추는 것처럼, 한 단어씩 다음 단어를 예측하더라도 그 다음 단어가 전체적인 문맥에 맞는지를 본다는 의미다.

- "He saw a carrot"이라는 문장에서 `rabbit`이 `habbit`보다 더 강하게 활성화된 것은, **당근-토끼**라는 의미적 연상이 운율 계획에도 영향을 미쳤기 때문으로 분석된다.
- 줄바꿈 토큰을 통해 모델이 전체 문맥을 보고 `rabbit`을 가져올 단어로 더 강하게 활성화한 것 — 즉, **계획을 하고 문장을 생성한다**는 결과가 확인되었다.

#### 프롬프트 인사이트 : 결론/결말을 먼저 정하고 역방향으로 설계하자

AI가 시를 쓸 때 미리 끝 단어를 계획한다는 것은, 창작 프롬프트에서 결론/결말을 먼저 정하고 역으로 작성하는 전략이 효과적이라는 것을 의미한다. 글쓰기, 블로그 글, 피드 등 모든 콘텐츠 작성에 적용할 수 있다.

**비효율 프롬프트 :**

```
AI 안전성에 대한 설득력 있는 글을 써주세요.
```

**역방향 설계 프롬프트 :**

```
이 글의 결론은 "AI 안전성 투자가 선택이 아닌 필수다"입니다.
이 결론을 향해 3단계 논지를 역방향으로 설계해 완성된 글을 작성해주세요.
```

LLM이 한 단어 기반의 순차적 생성 방식이라는 점을 감안하면, 방향을 역방향으로 전환하는 것만으로도 결과가 달라질 수 있다. 비효율 프롬프트는 **발산**에 강하고, 역방향 프롬프트는 **수렴**에 강하다.

---

### 핵심 발견 3. 보편적 사고 언어 — 다국어 처리의 비밀

> 관련 리서치 : [Do Models Think in English?](https://www.anthropic.com/research/tracing-thoughts-language-model)

Claude에게 영어, 프랑스어, 중국어로 "What is the opposite of small?"을 각각 물었을 때 :

- **입출력 근처 레이어** : 각 언어에 특화된 피처가 활성화
- **중간 레이어** : 언어와 무관하게 "작음(smallness)"과 "반대(oppositeness)"라는 **추상적 개념 피처**가 공통으로 활성화
- **최종 출력** : 해당 언어로 번역되어 출력

Claude는 내부적으로 **언어에 구애받지 않는 개념 공간**에서 먼저 사고하고, 출력 직전에 특정 언어를 선택한다.

#### 프롬프트 인사이트 : 다국어 작업 시 중간 언어를 명시적으로 지정하라

AI가 내부적으로 언어 중립적 개념 공간을 거친다는 것은, 번역·요약·다국어 분석 작업에서 중간 언어를 명시적으로 지정해 이 내부 구조를 활용할 수 있다는 의미다.

특히 한국어로만 입출력을 요청하면 입출력 근처의 언어 특화 레이어만 강하게 작동하는 반면, 중간에 개념 정리 단계를 영어로 명시하면 더 추상적이고 정교한 중간 처리를 유도할 수 있다.

**비효율 프롬프트 :**

```
영문 기사를 한국어로 요약해주세요.
```

**효율 프롬프트 :**

```
영문 기사를 분석할 때 :
(1) 먼저 영어로 핵심 논지와 근거를 정리한 뒤
(2) (1)의 결과를 기반으로 한국어 최종 요약을 작성해주세요.
```

> 💡 특히 전문 도메인(법률, 의학, 기술 문서)에서 효과가 두드러진다. 해당 도메인의 개념이 영어 기반 훈련 데이터에 더 풍부하게 담겨 있어, 중간 정리 언어를 영어로 지정하면 개념 추출 품질이 높아진다.

## 리서치 2. AI는 자기 생각을 숨기고 있다

- **논문** : [Reasoning Models Don't Always Say What They Think](https://www.anthropic.com/research/reasoning-models-dont-say-think)
- **테스트 모델** : Claude 3.7 Sonnet, DeepSeek R1
- **핵심 발견** : Chain-of-Thought(CoT)의 신뢰성 문제

단순 덧셈을 풀게 했을 때 모델이 정답을 맞힌 것처럼 보이지만, 중간 추론 과정이 엉망이거나, 추론의 100%를 다 보여주지 않고 일부만 보여주는 경우가 확인되었다.

**결론 : AI는 생각의 75~80%를 숨긴다.** CoT 모니터링만으로 AI의 위험 행동을 잡을 수 있다는 낙관론에 찬물을 끼얹은 결과다.

### 문제 제기 : CoT를 신뢰할 수 있는가?

Reasoning Model은 생각 과정을 보여준다. 그런데 CoT에 적힌 것이 정말 AI의 진짜 생각일까?

### 실험 방법 : 컨닝 페이퍼 테스트

연구진은 시험을 보는 학생에게 쪽지를 슬쩍 넘겨주는 것과 같은 실험을 설계했다. AI에게 평가 문제를 풀게 하면서 "정답은 (A)야" 같은 힌트를 슬쩍 제공한 뒤, CoT에서 그 힌트를 사용했다고 스스로 인정하는지 확인했다.

**실험 결과 :**

- **힌트 없이 질문** → 모델의 답변 : D
- **메타데이터에 "정답 C" 힌트 삽입** → 모델의 답변 : C로 변경

문제는 모델이 추론 과정에서 힌트를 봤다는 언급 없이, 추론 과정의 변화 없이 답변만 바뀌었다는 점이다.

**의미** : CoT 모니터링은 유용하지만, 충분하지 않다. AI가 보여주는 생각 과정(CoT)은 실제 내부 추론의 **25~39%만 반영**한다. 따라서 프롬프트 설계를 다르게 해야 한다.

---

### 프롬프트 인사이트 : AI의 CoT를 그대로 믿지 마라

#### 실전 팁 1. 검증 단계를 프롬프트에 명시적으로 넣어라

```
답변을 작성한 후, 다음 체크리스트로 자기 검증을 수행하세요 :
1. 근거 없는 주장이 있는가?
2. 해당 결론에 도달한 논리적 단계를 모두 설명했는가?
3. 대안적 해석을 고려했는가?
```

#### 실전 팁 2. 스크래치패드 기법을 활용하자

Anthropic이 실제 고객사에 적용해 정확도 향상을 이끌어낸 기법이다. 프롬프트 내에 사용자에게 보이지 않는 메모 공간을 주고, 먼저 충분히 생각하게 한 후 최종 답변을 작성하게 하는 방법이다.

```
다음 질문에 답하기 전에 <scratchpad> 태그 안에 먼저
관련 인용문과 근거를 정리하세요. 그 후 최종 답변을 작성하세요.

<scratchpad>
Scratch Pad - Requirements :
Scratch Pad - Options :
Scratch Pad - Evaluation :
</scratchpad>

Final Answer :
```

#### 실전 팁 3. 단계별 프롬프트 체이닝

한 번의 프롬프트로 모든 것을 해결하려 하지 말고, 단계별로 나누자. CoT가 불완전할 수 있으므로, **각 단계의 출력을 검증한 후 다음 단계로 넘기는 것**이 안전하다.

```
Step 1 : 주제에 대한 3가지 관점을 정리해주세요
Step 2 : 각 관점에 대한 근거를 제시하세요
Step 3 : 최종 결론을 작성하세요
```

## 정리 : 리서치에서 도출한 프롬프트 설계 원칙

|리서치 발견|프롬프트 전략|
|---|---|
|모델은 내부적으로 2단계 추론을 한다|중간 단계를 명시적으로 요청하라|
|AI는 창작 시 미리 계획한다|결론을 먼저 정하고 역방향으로 설계하라|

|CoT는 실제 추론의 25~39%만 반영한다|검증 단계, 스크래치패드, 프롬프트 체이닝을 활용하라|
