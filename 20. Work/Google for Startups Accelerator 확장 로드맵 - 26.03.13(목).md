## 1. 음성 인식과 실시간 처리

### 1-1. 미팅 중 실시간 코칭 — 끝나고 알려주는 게 아니라 하는 동안 도와주기

**현재 상태**
- 미팅 종료 후 전사 텍스트 기반 사후 분석 (질문 유형, 발화 비율, 주도권, 표현 패턴, 리더십 스코어)
- 모든 파이프라인이 배치 처리 기반으로 운영 중

**확장 방향**
- 미팅 중 실시간 코칭 힌트 제공: 리더 발화 과다 알림, 개방형 질문 부족 힌트, 멤버 발화 기회 부족 넛지, 민감 신호 감지 시 후속 질문 제안
- 사후 피드백 → 실시간 개입형 코칭으로 전환하여 행동 변화 효과 극대화

**필요 기술**
- 실시간 스트리밍 STT: 오디오 스트림을 수백ms 이내 텍스트로 변환 (gRPC 기반 양방향 스트리밍)
- 실시간 화자분리(diarization): 스트리밍 환경에서 2인 화자를 실시간 구분 — 배치보다 난이도가 훨씬 높음
- 온라인 대화 분석 엔진: 전체 트랜스크립트 없이 부분 문맥만으로 발화비율/질문유형/주도권을 판단하는 경량 분석 모델
- 저지연 LLM 추론: Gemini Flash 등 경량 모델을 활용한 실시간 코칭 메시지 생성 (latency 2초 이내 목표)
- 실시간 개입 시점 결정 모델: 언제 넛지를 주는 게 효과적이고 언제가 방해인지 판단하는 의사결정 로직
- 모델 경량화 및 엣지 배포: 클라이언트 측 경량 분석 모델로 latency 최소화 (TensorFlow Lite / MediaPipe 활용 가능)

**자체 한계**
- 실시간 스트리밍 처리 아키텍처 경험 전무 — 현재 모든 파이프라인이 배치 기반
- 실시간 STT도 어렵지만, 실시간 화자분리는 더 어려운 과제
- STT + 분석 + LLM 추천을 직렬로 연결하면 latency가 누적되어 실시간 제약을 맞추기 어려움
- 실시간 피드백은 틀리거나 늦으면 미팅 방해 — 사후 리포트보다 훨씬 높은 precision 요구
- 실시간 코칭 UI가 미팅 집중도를 해치지 않는 UX 설계 방법론 부족

**Google에서 받고 싶은 도움**
- Google Cloud Speech-to-Text V2 스트리밍 인식 경험 기반 실시간 STT 파이프라인 아키텍처 자문 (Chirp 3의 StreamingRecognize + 화자분리 활용)
- Gemini Flash를 활용한 저지연 실시간 추론 파이프라인 설계 코칭
- ~~TensorFlow Lite / MediaPipe를 활용한 클라이언트 측 경량 모델 배포 가이드~~
- 실시간 개입형 제품의 evaluation 기준 설계 지원 — Google Meet 등 실시간 제품 경험 기반

### 1-2. 자체 음성 인식 구축 — 외부 API에서 벗어나 직접 만들기

**현재 상태**
- AssemblyAI 기반 STT + 화자분리 운영 중 — 빠른 제품 출시에는 유리했으나 한계 체감 중
- Whisper 계열 모델(faster-whisper, WhisperX)로 자체 호스팅 실험 진행 중
- RunPod Serverless GPU 인프라 탐색 중

**확장 방향**
- 외부 API 의존도를 줄이고 자체 STT + 화자분리 파이프라인 확보
- 고객사별 커스터마이징(관련 단어 보정)
- 온프레미스/VPC 배포로 보안 민감 고객 대응
- 실시간 기능으로의 확장 기반 확보

**필요 기술**
- Whisper 기반 한국어 STT 품질 최적화: 도메인 용어(HR/리더십/코칭 관련 어휘) 적응, 음성 환경(회의실/원격) 대응
- 화자분리(diarization) 개선
- 발화 턴 세그멘테이션(turn segmentation): 화자 전환 시점을 정확히 잡아 분석 정확도 향상
- GPU 서빙 최적화: 콜드스타트 최소화, 오토스케일링, 비용 효율적 배치/스트리밍 처리
- 배치(batch) / 준실시간(near-real-time) / 실시간(real-time) 3단계 파이프라인 설계
- 모델 파인튜닝: 한국어 회의 데이터 기반 Whisper 모델 튜닝 파이프라인

**자체 한계**
- GPU 인프라 운영/최적화 경험 부재 — 현재까지 API 기반으로만 운영

**Google에서 받고 싶은 도움**

## 2. 리더와 멤버의 변화를 만드는 기능

### 2-1. 1:1 미팅 시뮬레이터 — 끝나고 피드백 받는 게 아니라 미리 연습하기

**현재 상태**

- 리더의 대화 행동 데이터와 약점 패턴 데이터가 세션별로 축적되는 구조
- 리더 유형 진단 (탐색형/업무집중형/코칭형/관계중심형)

### 2-2. 같은 리더, 다른 멤버 — 관계별 대화 패턴 분석

**현재 상태**
- 리더 개인 단위로 미팅 분석, 주제 주도권 분석(리더 vs 멤버) 구현 완료
- 동일 리더의 복수 멤버 세션 데이터가 축적 중

**확장 방향**
- 리더-멤버 조합(dyad) 단위 관계 패턴 분석: 같은 리더가 A 멤버와는 탐색형 질문이 많고, B 멤버와는 업무 지시형 발화가 많은 패턴 탐지
- 관계별 커뮤니케이션 스타일 차이, 라포 형성 정도, 심리적 안전감/신뢰 수준 변화, 반복되는 긴장 패턴 추적

**필요 기술**
- 동일 리더-멤버 pair를 장기 추적하는 dyadic timeline 데이터 모델
- 관계별 차이를 포착하는 interaction representation — 발화 패턴, 질문 스타일, 반응 속도 등 다차원 비교
- 신뢰/탐색/지시성/안전감의 변화 탐지 알고리즘
- sparse 데이터에서 과도한 해석을 피하는 confidence 설계 — 데이터 부족 시 '판단 보류' 로직

**자체 한계**
- 관계 다이나믹스를 텍스트 데이터로 모델링하는 학술적 프레임워크 부재
- 해석이 매우 민감 — '관계가 안 좋다' 식의 단정은 불가능하며 제품 메시지도 조심해야 함
- 데이터가 충분히 쌓여야 의미 있기 때문에 초기에는 강한 기능이라기보다 고급 인사이트 영역

**Google에서 받고 싶은 도움**
- 이 영역은 인프라 지원보다 모델링/표현 방식에 대한 자문이 더 중요
- 장기 대화 관계 모델링에 대한 기술적 피드백 — Google의 People Analytics 조직 연구 경험 기반
- 민감한 관계 신호를 다룰 때의 Responsible AI 가이드라인 수립 지원
- 신뢰도 기반 리포팅 설계 조언 — confidence score에 따라 표시 여부를 결정하는 UX 패턴
### 2-3. 조직 전체의 1:1 문화 진단 — 리더 한 명이 아니라 조직 전체 보기

**현재 상태**
- 리더 개인 단위 분석 중심
- 세션 메타데이터로 조직/팀/부서 정보 저장 중 (Supabase/PostgreSQL)

**확장 방향**
- 같은 조직의 여러 부서, 여러 리더 데이터를 집계하여 조직 수준 1:1 문화 진단
- 예: 전반적으로 업무 중심 대화에 치우쳐 있는지 / 멤버 발화 기회가 조직 전반에서 부족한지 / 코칭형 리더십이 특정 부서에만 나타나는지
- HR이 이를 바탕으로 추가 교육 대상 선정, 개입 필요 조직 식별, 교육 효과 측정 가능
- B2B SaaS로 계속 쓰이려면 리더 개인을 넘어 조직 전체 의사결정에 도움이 되어야 함

**필요 기술**
- 개인 분석 결과를 조직 단위로 안전하게 집계하는 pipeline — 익명화/privacy-preserving aggregation
- 소규모 그룹(팀 3~5명)에서 개인 식별 방지를 보장하는 k-anonymity / 차등 프라이버시 기술
- 부서별/직급별/조직규모별 벤치마크 세그먼트 분석
- 조직 리스크 탐지: 특정 팀에서 반복적으로 나타나는 부정적 패턴 자동 알림
- HR 담당자용 운영 대시보드 설계 — 조직 전체 뷰 + 드릴다운 상세

**자체 한계**
- 데이터 프라이버시 엔지니어링 경험 부재 — B2B 특성상 민감도가 매우 높음
- 소규모 집단에서 통계적 안정성 부족 — 팀 3명이면 집계 자체가 개인 식별 가능
- 자칫 감시 도구로 인식될 위험 — 제품 포지셔닝과 커뮤니케이션이 매우 중요
- HR 데이터 취급 관련 글로벌 규제(GDPR 등) 대응 경험 없음

**Google에서 받고 싶은 도움**
- Google의 프라이버시 엔지니어링 전문성 멘토링 — 차등 프라이버시, 연합학습(Federated Learning) 등 Google이 자체 제품에 적용해온 기술
- BigQuery 기반 조직 분석 데이터 파이프라인 아키텍처 자문 — 집계/세그먼트 분석에 BigQuery ML 활용
- Google Looker를 활용한 HR 의사결정용 대시보드 설계 가이드
- 엔터프라이즈 조직 분석 SaaS의 데이터 프라이버시 설계 베스트 프랙티스 (Google Workspace / People Analytics 사례)
- 소규모 집단 익명성 보장을 위한 기술적 가이드라인 수립 지원

## 3. 성과 데이터와 머신러닝

### 3-1. 좋은 미팅이 진짜 성과를 바꾸는가 — 비즈니스 임팩트 증명

**현재 상태**
- 미팅 텍스트/대화 패턴 분석으로 리더십 인사이트 제공
- 이것이 이직률, 직원 만족도, 성과 평가, 조직 몰입도와 어떻게 연결되는지는 아직 미증명

**확장 방향**
- 고객사 HR 데이터(이직률, 직원 만족도, 성과 평가)와 미팅 분석 데이터를 연결
- '좋은 1:1 미팅이 실제로 어떤 비즈니스 임팩트를 만드는가'를 정량적으로 증명
- 단순 '좋은 리포트'가 아니라 조직 성과와 연결된 리더십 운영 시스템으로 포지셔닝

**필요 기술**
- HRIS/성과/만족도 데이터 연동 파이프라인 — 고객사마다 다른 데이터 구조를 표준화하는 ETL
- 미팅 지표와 성과 지표 간 상관/인과 분석 — 인과 추론(causal inference) 방법론 적용
- 교란 변수 통제: 급여, 팀 규모, 산업, 재직 기간 등 미팅 외 변수 통제
- 설명 가능한 모델링: HR에게 '왜 이런 결과가 나왔는지' 설명할 수 있는 interpretable model
- ROI 대시보드: '1:1 미팅 품질 10% 향상 → 이직률 X% 감소' 정량 제시

**자체 한계**
- 인과 추론(causal inference) 방법론 경험 부재 — 상관관계와 인과관계 구분의 어려움
- 고객사 HR 시스템과의 데이터 통합 경험 없음 — 연동 난이도가 높음
- 인과관계로 과하게 해석하면 위험 — '미팅 품질이 이직률을 낮춘다'는 주장의 엄밀성 필요

**Google에서 받고 싶은 도움**
- BigQuery를 중심으로 한 HR 데이터 통합 아키텍처 자문 — 다양한 HRIS 시스템 데이터를 BigQuery로 통합하는 파이프라인 설계
- BigQuery ML을 활용한 상관/예측 분석 프레임 설계 — SQL 기반으로 로지스틱 회귀, 시계열 예측 등 직접 구축 가능
- Google People Analytics의 인과 추론 방법론 멘토링 — Google이 내부적으로 '관리자 행동 → 팀 성과' 관계를 어떻게 분석해왔는지
- Google Looker를 활용한 ROI 측정 대시보드 설계 베스트 프랙티스
- 성과와 AI 지표를 연결할 때의 책임 있는 커뮤니케이션 가이드 — 과대 해석 방지

### 3-2. 리더가 어떻게 성장하는지 예측하기 — 성장 예측 모델

**현재 상태**
- 리더의 경청 점수, 질문 습관, 발화 비율, 표현 패턴 등을 미팅별로 분석
- 아직은 '현재 상태 진단' 중심 — 시간에 따른 성장이나 피드백 효과 예측 수준은 아님
- 동일 리더의 복수 세션 스코어링 이력이 DB에 축적 중

**확장 방향**
- 리더의 경청 점수가 3개월간 어떻게 변했는지 추적
- 어떤 코칭 피드백을 받은 뒤 실제 행동 변화가 나타났는지 분석
- 어떤 유형의 리더는 어떤 피드백에 더 잘 반응하는지 예측
- 궁극적으로 '이 리더에게 어떤 개입을 하면 개선될 가능성이 높은가'를 보는 성장 예측 모델

**필요 기술**
- 종단(longitudinal) 데이터 구조 설계 — 동일 리더의 시간별 스코어 변화를 추적하는 시계열 데이터 모델
- 행동 변화 추적용 시계열 모델 — ARIMA, TimesFM 등을 활용한 스코어 트렌드 분석 및 예측
- 코칭 개입 효과 추적: '피드백 A를 받은 후 2주 내 점수가 변했는가' 같은 전후 비교 분석
- 리더 유형별 반응 차이를 보는 개인화 예측 모델 — 소규모 데이터에서의 과적합 방지 필수
- LLM 기반 진단 결과(정성적)를 ML 피처(정량적)로 변환하는 하이브리드 파이프라인

**자체 한계**
- ML 파이프라인 설계 경험 부재 — 시계열 분석/예측 모델 전문성 없음
- 장기 데이터 부족 — 아직 충분한 종단 데이터가 쌓이지 않은 초기 단계
- 실제 코칭 개입과 변화 사이의 라벨(ground truth) 부족
- 점수 체계의 안정성과 예측 가능성 검증 필요

**Google에서 받고 싶은 도움**
- BigQuery ML의 ARIMA_PLUS / TimesFM 모델을 활용한 시계열 예측 설계 자문 — SQL만으로 시계열 예측 모델 구축 가능
- Vertex AI AutoML을 활용한 리더 유형별 개인화 예측 모델 구축 가이드
- LLM 기반 진단 결과를 ML 피처로 전환하는 하이브리드 파이프라인 설계 멘토링
- 소규모 데이터 환경에서의 과적합 방지 및 유의미성 검증 방법론
- longitudinal ML 평가 방식 코칭 — 시간에 따른 모델 성능 변화 추적

### 3-3. 이 멤버, 떠날 수도 있습니다 — 이탈 리스크 조기 탐지

**현재 상태**
- 현재 트랜스크립트 기반으로 LLM이 멤버 위험 신호를 판단하려 하지만 정확도 부족
- 실제 탐지 성능이 낮아 프로덕션 수준이 아닌 상태

**확장 방향**
- 성과 데이터가 연결되면 훨씬 근거 있는 접근 가능
- 멤버의 발화 패턴 + 리더의 커뮤니케이션 성향 + 특정 대화 스크립트 + 반복 표현/침묵/회피 패턴을 피처로 활용
- 실제로 이직/성과 저하/만족도 하락/disengagement가 일어났는지와 연결하여 근거 기반 ML 모델 구축
- LLM 직감형 탐지 → 데이터 기반 예측 모델로 전환

**필요 기술**
- outcome-linked dataset 구축: 미팅 데이터 + 실제 이직/성과/만족도 결과를 연결한 학습 데이터
- 멤버 리스크 정의 체계: '이탈 리스크'를 어떤 기준으로 정의하고 라벨링할 것인지
- LLM 신호 + 구조화 피처를 결합한 hybrid model — LLM이 추출한 정성적 신호(감정, 회피, 불만)를 정량 피처와 결합
- false positive를 극도로 낮추는 보수적 리스크 모델 — 오탐이 치명적인 영역
- 설명 가능성(explainability): HR에게 '왜 이 멤버가 리스크인지' 근거를 제시할 수 있어야 함
- 노무/윤리/프라이버시 검토: 직원 이탈 예측은 법적/윤리적으로 매우 민감한 영역

**자체 한계**
- 실제 outcome 데이터 없이는 모델 학습 자체가 불가능 — 고객사 HR 데이터 연동이 전제
- 민감도가 극히 높은 영역 — 오탐 시 조직 신뢰 훼손, 직원 관계 악화 위험
- 노무/윤리/프라이버시 이슈가 크고, 국가별 규제가 다름
- LLM + ML 하이브리드 모델 설계 경험 없음

**Google에서 받고 싶은 도움**
- HR outcome prediction에 대한 ML 프레임 설계 자문 — BigQuery ML의 로지스틱 회귀/부스티드 트리 모델 활용 가이드
- 민감한 사람 관련 예측 모델의 Responsible AI 가이드 — Google의 AI 원칙(AI Principles) 기반 윤리적 프레임워크
- Vertex AI의 Explainable AI(XAI) 기능을 활용한 모델 해석 가능성 확보 방법
- LLM 신호를 ML 피처로 전환하는 hybrid modeling 전략 자문
- Google People Analytics의 직원 리스크 분석 경험에서 배우는 보수적 모델 설계 원칙
- false positive 최소화를 위한 threshold 설계 및 단계적 알림 체계 가이드
